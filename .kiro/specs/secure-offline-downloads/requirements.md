# Requirements Document

## Introduction

This feature enables students to securely download course lectures (videos) and study materials (PDFs) for offline access within the app on native platforms (iOS and Android) only. Downloads are stored in app-private, encrypted storage — hidden from the device gallery and file manager — and are only accessible through the existing Downloads screen at `app/downloads.tsx` (accessible from the student profile). The system enforces automatic deletion of downloaded content when access conditions are no longer met (course validity expiry, course removal, student blocking, student removal, or unenrollment). Web users access content online only; the download feature is not available on the web platform.

The app targets iOS and Android (React Native + Expo). The backend is Express + PostgreSQL (Neon), with files hosted on Cloudflare R2. The existing `user_downloads` table, `download_allowed` columns on `lectures` and `study_materials`, and `app/downloads.tsx` page (already wired to `/api/my-downloads` with Lectures/PDFs tabs) are already in place and will be extended — not replaced — by this feature.

---

## Glossary

- **Download_Manager**: The client-side service responsible for downloading, encrypting, storing, and deleting files on the device. Runs on native platforms only.
- **Download_Button**: The UI control shown on a lecture or study material when `download_allowed = true` and the platform is native (iOS or Android).
- **Downloads_Section**: The existing screen at `app/downloads.tsx` (accessible from the student profile) where students access their downloaded content. This screen already has Lectures/PDFs tabs and fetches from `/api/my-downloads` — it will be extended to show offline availability status and support local file playback.
- **Secure_Storage**: App-private, encrypted file storage inside the app's `documentDirectory` (Expo FileSystem), not accessible outside the app, the device gallery, or the file manager.
- **Signed_URL**: A time-limited (30 seconds), single-use, authenticated URL generated by the backend to authorize a single proxied download.
- **Download_Proxy**: The backend endpoint that streams file content to the client, replacing direct R2 URL access. Adds a user-specific watermark token to response headers.
- **Watermark_Token**: A user-specific identifier embedded in download response headers by the Download_Proxy, enabling leak tracing.
- **Obfuscated_Filename**: A UUID-based filename assigned to a downloaded file on disk, replacing the original filename to prevent content identification if the device is compromised.
- **Encryption_Key**: A per-user or per-device secret key used to encrypt downloaded files at rest using AES-256.
- **Auto_Deletion**: The process of removing locally stored files and their `user_downloads` records when access conditions are no longer met.
- **Enrollment**: A record linking a student to a course, optionally with a validity expiry date.
- **Course_Validity**: An optional expiry date on an enrollment after which the student loses access to course content.
- **Student**: A user with `role = 'student'` in the `users` table.
- **Admin**: A user with `role = 'admin'` in the `users` table.
- **Lecture**: A video-based course item in the `lectures` table with a `download_allowed` boolean column.
- **Lecture_File**: The video file associated with a Lecture, stored on Cloudflare R2.
- **Study_Material**: A PDF-based course item in the `study_materials` table with a `download_allowed` boolean column.
- **Material_File**: The PDF file associated with a Study_Material, stored on Cloudflare R2.

## Requirements

### Requirement 1: Display Download Button (Native Only)

**User Story:** As a student on a mobile device, I want to see a download button on lectures and study materials that allow downloading, so that I know which content I can save for offline use.

#### Acceptance Criteria

1. WHEN a student views a Lecture with `download_allowed = true` and the student is enrolled in the course and the platform is iOS or Android, THE Download_Button SHALL be displayed in the following locations:
   - On the lecture row inside the course detail screen (`app/course/[id].tsx`)
   - On the lecture player screen (`app/lecture/[id].tsx`)
2. WHEN a student views a Study_Material with `download_allowed = true` and the student is enrolled in the course and the platform is iOS or Android, THE Download_Button SHALL be displayed in the following locations:
   - On the material row inside the course detail screen (`app/course/[id].tsx`)
   - On the material viewer screen (`app/material/[id].tsx`)
   - On the material row inside the material folder screen (`app/material-folder/[name].tsx`)
3. WHEN the platform is web (`Platform.OS === 'web'`), THE Download_Button SHALL NOT be rendered anywhere — not in course detail, lecture player, material viewer, or material folder screens.
4. WHEN a Lecture has `download_allowed = false`, THE Download_Button SHALL NOT be rendered for that Lecture in any screen.
5. WHEN a Study_Material has `download_allowed = false`, THE Download_Button SHALL NOT be rendered for that Study_Material in any screen.
6. WHEN a student has already downloaded a Lecture or Study_Material, THE Download_Button SHALL display a visual indicator (e.g., checkmark or "Downloaded" label) in all locations where it appears.

### Requirement 2: Secure File Download via Backend Proxy

**User Story:** As a student, I want to download allowed content to my device, so that I can access it offline without an internet connection.

#### Acceptance Criteria

1. WHEN a student taps the Download_Button for a Lecture or Study_Material, THE Download_Manager SHALL request a Signed_URL from the backend before initiating the file transfer.
2. WHEN the backend receives a signed URL request, THE Backend SHALL verify that the requesting student is enrolled in the course containing the requested item before issuing the Signed_URL.
3. IF the student is not enrolled in the course, THEN THE Backend SHALL return a 403 error and THE Download_Manager SHALL display an access-denied message to the student.
4. WHEN a Signed_URL is obtained, THE Download_Manager SHALL download the file exclusively through the Download_Proxy endpoint; THE Download_Manager SHALL NOT use direct R2 URLs.
5. THE Backend SHALL generate Signed_URLs with a maximum validity of 30 seconds to limit unauthorized reuse.
6. THE Backend SHALL mark a Signed_URL as used after the first download attempt so that the Signed_URL cannot be reused for a second download.
7. IF a Signed_URL has already been used, THEN THE Backend SHALL return a 403 response when the Signed_URL is presented again.
8. WHEN the Download_Proxy streams a file to the client, THE Download_Proxy SHALL add a Watermark_Token header containing a user-specific identifier to the response.
9. WHEN a file is received by the Download_Manager, THE Download_Manager SHALL encrypt the file content using AES-256 before writing it to Secure_Storage.
10. THE Download_Manager SHALL store encrypted files using Obfuscated_Filenames (UUID-based) in the app's `documentDirectory`, ensuring the original filename is never written to disk.
11. WHEN a file download completes successfully, THE Download_Manager SHALL record the download in the `user_downloads` table via `POST /api/my-downloads` with `item_type`, `item_id`, and the Obfuscated_Filename.
12. WHEN a file download is in progress, THE Download_Button SHALL display a progress indicator showing download percentage.
13. IF a file download fails due to a network error, THEN THE Download_Manager SHALL display an error message to the student and SHALL NOT create a `user_downloads` record for the failed download.

### Requirement 3: Offline Playback and Viewing

**User Story:** As a student, I want to open downloaded lectures and study materials from the Downloads section without an internet connection, so that I can study anywhere.

#### Acceptance Criteria

1. WHEN a student opens the Downloads_Section, THE Downloads_Section SHALL display all downloaded Lectures and Study_Materials grouped by type (videos and PDFs).
2. WHEN a student taps a downloaded Lecture in the Downloads_Section, THE Download_Manager SHALL decrypt the file from Secure_Storage and serve the video using a local file URI rather than a remote URL.
3. WHEN a student taps a downloaded Study_Material in the Downloads_Section, THE Download_Manager SHALL decrypt the file from Secure_Storage and serve the PDF using a local file URI rather than a remote URL.
4. WHILE the device has no internet connection, THE Downloads_Section SHALL remain fully functional for previously downloaded content.
5. WHEN a downloaded file is not found in Secure_Storage (e.g., cleared by OS), THE Downloads_Section SHALL display a "File not available" message for that item and SHALL offer the student the option to re-download it.

---

### Requirement 4: Encrypted Private Storage

**User Story:** As a product owner, I want downloaded content to be encrypted and inaccessible outside the app, so that course content is protected from unauthorized redistribution even on rooted or jailbroken devices.

#### Acceptance Criteria

1. THE Download_Manager SHALL store all downloaded files inside the app's private `documentDirectory` on iOS and Android, ensuring the files do not appear in the device gallery or file manager.
2. THE Download_Manager SHALL encrypt all downloaded file content using AES-256 before writing to disk, using an Encryption_Key that is not stored in plaintext on the filesystem.
3. THE Download_Manager SHALL assign each downloaded file an Obfuscated_Filename (UUID-based) so that the file's content or course association cannot be determined from the filename alone.
4. THE Download_Manager SHALL NOT use `cacheDirectory` or any publicly accessible path for storing downloaded content.
5. THE Backend SHALL NOT expose direct, unauthenticated R2 file URLs to the client; all file access SHALL be mediated through the Download_Proxy.

---

### Requirement 5: Screenshot and Screen Recording Prevention

**User Story:** As a product owner, I want to prevent screenshots and screen recordings of downloaded video content, so that students cannot capture and redistribute protected material.

#### Acceptance Criteria

1. WHEN a downloaded Lecture video is playing on Android, THE Download_Manager SHALL set the `FLAG_SECURE` flag on the video player window to prevent screenshots and screen recordings.
2. WHEN a downloaded Lecture video is playing on iOS, THE Download_Manager SHALL apply the platform secure flag equivalent to prevent screenshots and screen recordings.
3. WHEN the video player is closed or the student navigates away from the playback screen, THE Download_Manager SHALL remove the secure window flag to restore normal screenshot behavior for other screens.

---

### Requirement 6: Auto-Deletion on Access Revocation

**User Story:** As a product owner, I want downloaded content to be automatically deleted when a student's access is revoked, so that students cannot retain content they are no longer entitled to.

#### Acceptance Criteria

1. WHEN a student's enrollment in a course is deleted or the student is unenrolled by an Admin, THE Auto_Deletion process SHALL delete all locally stored files for that course from Secure_Storage and SHALL remove the corresponding `user_downloads` records for that student and course.
2. WHEN a course is deleted by an Admin, THE Auto_Deletion process SHALL delete all locally stored files for that course from Secure_Storage for all affected students and SHALL remove the corresponding `user_downloads` records.
3. WHEN a student's account is blocked by an Admin (`is_blocked = true`), THE Auto_Deletion process SHALL delete all locally stored files from Secure_Storage for that student and SHALL remove all of that student's `user_downloads` records.
4. WHEN a student's account is deleted from the system, THE Auto_Deletion process SHALL delete all locally stored files from Secure_Storage for that student (cascade delete on `user_downloads` via the existing `ON DELETE CASCADE` foreign key is sufficient for the database records).
5. WHEN a student's Course_Validity expires (enrollment expiry date is reached), THE Auto_Deletion process SHALL delete all locally stored files for that course from Secure_Storage and SHALL remove the corresponding `user_downloads` records for that student and course.
6. WHEN the app is brought to the foreground, THE Download_Manager SHALL check all downloaded items against current enrollment and access status on the server and SHALL trigger Auto_Deletion for any items whose access has been revoked before rendering the Downloads_Section.
7. IF the Auto_Deletion process cannot delete a file from Secure_Storage due to a filesystem error, THEN THE Download_Manager SHALL log the error and SHALL mark the `user_downloads` record as `deletion_pending` so that deletion is retried on the next foreground event.

### Requirement 7: Extend Existing Downloads Screen

**User Story:** As a student, I want the existing Downloads section in my profile to show which content is available offline and let me open it without internet, so that I can manage my offline files in one place.

#### Acceptance Criteria

1. THE Downloads_Section SHALL continue to fetch from `GET /api/my-downloads` and display items in the existing Lectures/PDFs tabs — no new screen shall be created.
2. WHEN the `GET /api/my-downloads` response contains a Lecture or Study_Material whose local encrypted file is present in Secure_Storage, THE Downloads_Section SHALL show a green "Available Offline" badge on that item.
3. WHEN the `GET /api/my-downloads` response contains a Lecture or Study_Material whose local file is NOT present in Secure_Storage, THE Downloads_Section SHALL show a "Re-download" button for that item.
4. WHEN a student taps a downloaded item in the Downloads_Section, THE Downloads_Section SHALL decrypt and open the item using the local file URI if available, or fall back to the remote URL if the local file is missing.
5. THE Downloads_Section SHALL display the total storage used by downloaded files in a summary line at the top of the screen.
6. WHEN a student long-presses or swipes a downloaded item, THE Downloads_Section SHALL present a "Delete Download" option that removes the local encrypted file from Secure_Storage and deletes the `user_downloads` record via `DELETE /api/my-downloads/:itemType/:itemId`.

### Requirement 8: Backend Download Authorization and Proxy Endpoint

**User Story:** As a backend developer, I want a secure proxy endpoint that issues time-limited single-use signed URLs and streams file content, so that only enrolled students can download files and all access is auditable.

#### Acceptance Criteria

1. THE Backend SHALL expose `GET /api/download-url?itemType=lecture|material&itemId=:id` as an authenticated endpoint that returns a Signed_URL valid for 30 seconds.
2. WHEN a valid authenticated request is received, THE Backend SHALL verify enrollment and `download_allowed = true` before generating a Signed_URL.
3. IF `download_allowed = false` for the requested item, THEN THE Backend SHALL return a 403 response with a descriptive error message.
4. IF the student is not enrolled in the course containing the requested item, THEN THE Backend SHALL return a 403 response with a descriptive error message.
5. THE Backend SHALL expose `GET /api/download-proxy?token=:signedToken` as the Download_Proxy endpoint that streams the file from R2 to the client.
6. WHEN the Download_Proxy handles a request, THE Backend SHALL mark the Signed_URL token as used in the database before streaming the file.
7. WHEN the Download_Proxy streams a file, THE Backend SHALL include a `X-Watermark-Token` response header containing a user-specific identifier derived from the student's ID and the current timestamp.
8. IF the Signed_URL token presented to the Download_Proxy has already been used or has expired, THEN THE Backend SHALL return a 403 response and SHALL NOT stream any file content.
9. THE Backend SHALL expose `DELETE /api/my-downloads/:itemType/:itemId` as an authenticated endpoint that removes the `user_downloads` record for the requesting student.
10. WHEN an Admin deletes a course or unenrolls a student, THE Backend SHALL call an internal cleanup function that removes all `user_downloads` records for the affected student-course combination.
11. WHEN an Admin blocks a student, THE Backend SHALL remove all `user_downloads` records for that student.

---

### Requirement 9: Enrollment Validity Expiry Support

**User Story:** As an admin, I want to set an expiry date on a student's enrollment, so that downloaded content is automatically removed when the course access period ends.

#### Acceptance Criteria

1. THE Backend SHALL support an optional `valid_until` timestamp column on the `enrollments` table representing the date after which the enrollment is considered expired.
2. WHEN `GET /api/my-downloads` is called, THE Backend SHALL exclude items from courses where the student's enrollment `valid_until` has passed.
3. WHEN `GET /api/download-url` is called for an item in a course where the student's enrollment `valid_until` has passed, THE Backend SHALL return a 403 response.
4. THE Backend SHALL expose an admin endpoint `PUT /api/admin/enrollments/:id` that allows setting or updating the `valid_until` field on an enrollment.
